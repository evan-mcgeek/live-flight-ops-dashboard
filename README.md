# Live Flight Ops Dashboard

A showcase project: live aircraft tracking on a map, in a searchable list,
and in detail — a Flutter mobile frontend backed by a lightweight .NET 8
API, both built with heavy AI assistance (see below).

## Layout

```
live_flight_ops_dashboard/
├── app/    # Flutter mobile frontend — see app/README.md
├── api/    # .NET 8 backend (REST + SignalR) — see api/README.md
└── .ai/    # everything AI-related — see below
```

## Running both together

1. Start the API first:
   ```bash
   cd api/src/FlightOps.Api
   dotnet run
   ```
   Listens on `http://localhost:5273` by default.
2. Then the app, pointed at it (the default already matches step 1, so
   this is only needed for a different host):
   ```bash
   cd app
   flutter pub get && dart run build_runner build --delete-conflicting-outputs
   flutter run --dart-define=API_BASE_URL=http://localhost:5273
   ```

See `app/README.md` and `api/README.md` for full setup/test instructions.

## API credentials

The API needs an OpenSky Network OAuth2 client id/secret in a gitignored
`api/src/FlightOps.Api/appsettings.secrets.json` — see `api/README.md`'s
credentials section for the exact shape and where to register a client.

## `.ai/` — everything AI-related lives here

If you're an agent (or a person) picking up work on this repo, `.ai/` at
the repo root is the single place to look for anything AI-produced or
AI-tooling-related — nothing meaningful of this kind is scattered in
`app/` or `api/`.

```
.ai/
├── .understand-anything/        # knowledge graphs: app/, api/, and merged
├── .claude/
│   ├── app-skills/               # lessons learned building the app
│   └── api-skills/               # lessons learned building the API
├── docs/superpowers/
│   ├── specs/                    # current design docs (backend + frontend)
│   └── plans/                    # build retrospectives + this finalization plan
└── design/                       # visual mockups that drove the app's UI
```

- **Knowledge graphs** (`.ai/.understand-anything/`) — architecture layers,
  file relationships, and a guided tour, generated by the
  `understand-anything` Claude Code plugin. Three graphs: `app/` alone,
  `api/` alone, and a merged one covering both. To browse the merged one
  interactively, first link it to where the dashboard expects it (one-time,
  not committed — it's local tooling convenience, not project content):
  ```bash
  ln -s .ai/.understand-anything .understand-anything
  ```
  then, from repo root:
  ```
  /understand-dashboard .
  ```
- **Skills** (`.ai/.claude/{app,api}-skills/`) — project-specific
  patterns and gotchas captured while building this project (bloc
  lifecycle rules, SignalR reconnect handling, codegen staleness
  pitfalls, etc.). These used to live at `app/.claude/skills/` and
  `api/.claude/skills/`, where Claude Code auto-discovers and applies
  them by default; they were consolidated here for a single point of
  reference, which means they're no longer auto-triggered by that
  discovery mechanism — read them directly if you want the lessons.
- **Superpowers docs** (`.ai/docs/superpowers/`) — `specs/` holds the
  current design for each half of the project; `plans/` holds condensed
  build retrospectives (not step-by-step task logs) plus this
  finalization plan.
- **Design** (`.ai/design/`) — the original HTML/JSX mockups the app's
  visual design was built from.

## Claude Code plugins used to build this project

This project was built with help of Claude Code, using several
of its plugins beyond the base agent loop:

- **[superpowers](https://claude.com/plugins/superpowers)** — a
  structured workflow toolkit: brainstorming a design before writing code,
  turning an approved design into a task-by-task implementation plan,
  executing that plan (one fresh subagent per task, reviewed as it goes),
  and requesting a final code review before calling a phase done. Both
  `app/` and `api/` were designed and built this way — see
  `.ai/docs/superpowers/{specs,plans}/`.
- **[understand-anything](https://understand-anything.com/)** —
  generates an interactive knowledge graph of a codebase (architecture
  layers, file/function relationships, a guided tour) and a local
  dashboard to browse it. Used to produce everything under
  `.ai/.understand-anything/`.
- **autoharness** — distills finished work sessions into reusable
  project-specific skills (patterns, gotchas, conventions worth
  remembering) rather than losing that knowledge at the end of each
  session. This is how the lessons under `.ai/.claude/{app,api}-skills/`
  came to exist in the first place.
- **ponytail** — a lazy-by-design behavioral mode (YAGNI-first, prefer the
  standard library/an existing pattern over new code, shortest working
  diff) used during parts of this project's implementation and cleanup
  passes to keep the codebase from accumulating unnecessary abstraction.
